INTRODUCCIÓN
La transformación digital en el ámbito educativo ha impulsado la necesidad de contar con plataformas tecnológicas que faciliten el acceso a recursos académicos de manera rápida, organizada y accesible desde cualquier lugar. En este contexto, las bibliotecas digitales se han convertido en una herramienta fundamental para estudiantes, docentes e investigadores, ya que permiten consultar y gestionar contenido bibliográfico sin las limitaciones de una biblioteca física, optimizando así los procesos de aprendizaje y acceso a la información.
El presente documento describe el desarrollo de un Sistema de Biblioteca Digital, una plataforma diseñada para gestionar de manera eficiente libros electrónicos, préstamos, devoluciones y reservas dentro de un entorno digital. El sistema permitirá organizar el catálogo de libros y ofrecer acceso a diferentes formatos de lectura como PDF, EPUB y MOBI, facilitando su consulta desde distintos dispositivos. Asimismo, la plataforma incorporará un modelo de suscripciones y un módulo de recomendaciones que mejorará la experiencia del usuario mediante sugerencias basadas en sus intereses y comportamiento de lectura.
Además, el proyecto contempla la aplicación de principios de arquitectura moderna y buenas prácticas de desarrollo de software con el objetivo de construir un sistema escalable, mantenible y capaz de soportar acceso simultáneo de múltiples usuarios. Asimismo, durante su desarrollo se incorporará la aplicación de patrones de diseño (GoF), de esta manera, el sistema busca ofrecer una solución tecnológica robusta que contribuya a la gestión eficiente de recursos bibliográficos en entornos educativos y académicos.
 
DESCRIPCIÓN DEL PROYECTO
El Sistema de Biblioteca Digital es una plataforma tecnológica orientada al ámbito educativo y académico, diseñada para facilitar el acceso, la gestión y la distribución de contenido bibliográfico digital de forma eficiente, segura y escalable. Su objetivo es centralizar la administración de recursos digitales y ofrecer a estudiantes, docentes e investigadores un entorno que permita consultar, prestar y gestionar libros electrónicos desde cualquier dispositivo con conexión a internet. La plataforma permitirá administrar el catálogo digital mediante el registro, clasificación y organización de libros en diferentes áreas del conocimiento, además de gestionar procesos de préstamos, devoluciones y reservas.
El sistema será compatible con múltiples formatos de lectura como PDF, EPUB y MOBI, lo que permitirá a los usuarios acceder al contenido desde distintos dispositivos y aplicaciones de lectura. Asimismo, incorporará un modelo de suscripciones para gestionar diferentes niveles de acceso y beneficios, junto con un módulo de recomendaciones personalizadas que sugerirá libros según el comportamiento y las preferencias de cada usuario.
Desde el punto de vista técnico, el sistema estará diseñado para soportar acceso multiusuario simultáneo, aplicando principios de arquitectura moderna y buenas prácticas de desarrollo de software que garanticen un alto nivel de rendimiento, escalabilidad y mantenibilidad de la plataforma.

ARQUITECTURA HEXAGONAL
El proyecto de la Biblioteca Digital está organizado siguiendo el modelo de arquitectura hexagonal o Ports and Adapters, el cual permite separar claramente la lógica de negocio del código dependiente de frameworks y tecnologías externas. La aplicación se divide en tres capas principales: Domain, Application e Infrastructure. El dominio representa el núcleo del sistema y las demás capas dependen de él, garantizando que la lógica de negocio permanezca aislada de detalles técnicos como bases de datos, frameworks o protocolos de comunicación.
Capa de Dominio (Domain)
La capa domain contiene el modelo de negocio y la lógica central del sistema. Dentro de esta se encuentran las carpetas model, donde se definen las entidades principales del sistema; port.in, que contiene los puertos de entrada o contratos de los casos de uso; y port.out, donde se definen los puertos de salida utilizados para la persistencia de datos.
También incluye la carpeta service, donde se ubican servicios de dominio como la gestión de subida de archivos, organizada en subcarpetas como upload.factory, upload.factory.creators y upload.uploaders, que implementan la lógica para manejar diferentes formatos de archivos. Además, existe la carpeta subscription, encargada de la gestión relacionada con suscripciones dentro del sistema.
Capa de Aplicación (Application)
La capa application contiene los servicios de aplicación responsables de orquestar los casos de uso del sistema. En esta capa se encuentra la carpeta service, donde se ubican los servicios que coordinan la interacción entre los puertos definidos en el dominio y los adaptadores de infraestructura, como BookService, PrestamoService, RecomendacionService, SubscriptionService y UserService.
Capa de Infraestructura (Infrastructure)
La capa infrastructure contiene los adaptadores que conectan el sistema con tecnologías externas. Dentro de adapter.in.web se encuentran los controladores que exponen la API REST, como BookController, PrestamoController, RecomendacionesController y UserController. Por otro lado, adapter.out.persistence contiene la integración con la base de datos, organizada en las subcarpetas adapter, entity y repository. Finalmente, la carpeta config incluye configuraciones generales de la aplicación, como BeanConfiguration y GlobalExceptionHandler.

PATRÓN SINGLETON (ENUM)
El Singleton es un patrón de diseño creacional que garantiza que una clase tenga una única instancia y proporcione un punto de acceso global a la misma, tal como lo describen GeeksforGeeks y Refactoring.Guru al enfatizar la “single instance” y el “global access point”. Según GeeksforGeeks, este patrón se utiliza cuando se necesita control centralizado sobre recursos compartidos como configuraciones, logging o conexiones a base de datos, donde múltiples instancias serían costosas o inconsistentes. Refactoring.Guru y Amit Bhardwaj coinciden en que su intención principal es controlar el acceso a un recurso compartido y ofrecer un punto de acceso global bien definido, lo que explica sus distintas variantes:
	Eager Initialization
	Lazy Initialization (thread-safe)
	Double-Checked Locking
	Enum Singleton (recomendado por Bloch).

•	Código implementado
 

•	Justificación de la implementación
Se implementó el patrón Singleton basado en enum. Amit Bhardwaj destaca que el Enum Singleton en Java es una de las formas más simples y seguras de implementar un Singleton, ya que la JVM garantiza una única instancia por constante enum, protección frente a problemas de reflexión y serialización, y thread-safety por diseño. Siguiendo esa recomendación, la implementación de SubscriptionManager.INSTANCE se apoya en las ventajas del Enum Singleton descritas por Bhardwaj: código minimalista (una sola constante INSTANCE sin constructores privados ni métodos getInstance()), inicialización controlada por la JVM y comportamiento robusto en entornos concurrentes. Dado que, como explican tanto Refactoring.Guru como GeeksforGeeks, el Singleton es especialmente adecuado para componentes de configuración o recursos globales, modelar SubscriptionManager como un Enum Singleton encaja de forma natural con un contexto de suscripciones donde la lógica de negocio es global, stateless y debe ser consistente para todos los consumidores del sistema.
En el contexto específico de suscripciones, SubscriptionManager encapsula lógica de negocio pura que debe ser consistente globalmente sin variaciones por request o usuario, haciendo del Singleton Enum la elección técnicamente superior para este dominio stateless y de consulta única.

PATRÓN FACTORY METHOD
El Factory Method es un patrón de diseño creacional cuyo propósito es definir una interfaz o método para crear objetos, pero dejar que las subclases decidan qué clase concreta deben instanciar, tal como lo describe GeeksforGeeks al destacar que "el cliente no necesita saber qué clase concreta se está instanciando". En lugar de usar directamente un constructor (new), el cliente llama a un método fábrica que encapsula la creación de objetos, promoviendo desacoplamiento y extensibilidad del código, según explica Ashish Pratap en AlgoMaster al enfatizar su rol en sistemas escalables. Se usa principalmente para:
	Evitar acoplar el código cliente a clases concretas de objetos.
	Permitir que nuevas variantes de objetos se integren fácilmente sin modificar el cliente.
	Encapsular la lógica de creación compleja en una sola parte del código.
	Esto facilita seguir principios sólidos de diseño como Open/Closed (abierto para extensión, cerrado para modificación).

•	Justificación de la implementación
El sistema de biblioteca digital debe permitir la carga de libros en distintos formatos:
	PDF
	EPUB
	MOBI
Cada formato requiere una lógica de procesamiento distinta.
Si se implementará directamente en el servicio algo como:
if (formato == PDF) { ... }
else if (formato == EPUB) { ... }
else if (formato == MOBI) { ... }
Se generarían los siguientes problemas, tal como advierte GeeksforGeeks sobre el uso de condicionales extensa:
	Alto acoplamiento
	Violación del principio Open/Closed
	Dificultad para agregar nuevos formatos
	Código difícil de mantener
Ashish Pratap coincide en que esta aproximación genera "código rígido y difícil de mantener" al dificultar la adición de nuevos formatos. Por lo cual se utilizó el patrón Factory Method para encapsular la creación de objetos relacionados con la carga de archivos según su formato. Esta decisión permite desacoplar la lógica de negocio de la lógica de instanciación, reducir el uso de condicionales y cumplir con el principio Open/Closed. La implementación, apoyada por el contenedor IoC de Spring, garantiza extensibilidad, mantenibilidad y coherencia arquitectónica dentro del sistema.

PATRÓN ABSTRACT FACTORY
El Abstract Factory es un patrón de diseño creacional que proporciona una interfaz para crear familias de objetos relacionados o dependientes sin especificar sus clases concretas, tal como lo define Refactoring.Guru al describirlo como un patrón que "produce familias de objetos relacionados sin especificar sus clases concretas". Esto permite que el cliente utilice distintos conjuntos de objetos compatibles entre sí sin quedar acoplado a sus implementaciones específicas, promoviendo consistencia y extensibilidad. TutorialsPoint explica que proporciona un nivel superior de abstracción para la creación de objetos, asegurando que los productos de una familia funcionen juntas correctamente.
•	Código implementado
 
 

•	Justificación de la implementación
En el desarrollo del sistema se implementó el patrón Abstract Factory en conjunto con Factory Method para gestionar la creación de objetos de manera flexible y desacoplada, como recomienda Refactoring.Guru para escenarios con familias de productos intercambiables. El Abstract Factory se encargó de definir una interfaz para crear familias de objetos relacionados (por ejemplo, uploaders para formatos de libros), permitiendo que el cliente utilice diferentes implementaciones sin depender de clases concretas.

PATRÓN BUILDER (FLUENT BUILDER)
El Builder es un patrón de diseño creacional que permite construir objetos complejos paso a paso, separando la construcción de su representación final para evitar constructores telescópicos con múltiples parámetros opcionales. Refactoring.Guru explica que organiza la construcción en pasos discretos (como setTitle(), setAuthor(), build()), permitiendo producir diferentes configuraciones de un objeto usando el mismo código de construcción, con un director opcional para definir secuencias específicas. TutorialsPoint destaca que encapsula la lógica de creación compleja en un objeto Builder separado, facilitando la creación de diferentes representaciones del mismo objeto sin modificar el cliente.

•	Código implementado
 
 
 
 

•	Justificación de la implementación
Se implementó el patrón Builder en la clase Préstamo debido a su alta complejidad estructural, caracterizada por 13 atributos donde solo 3 (usuarioId, libroId, fechaPrestamo) son obligatorios y el resto son opcionales (como fechaDevolucionReal, multasAcumuladas, motivoRechazo). Esta configuración generaba el clásico "constructor telescópico", donde múltiples constructores sobrecargados complicaban la legibilidad y aumentaban el riesgo de errores por orden incorrecto de parámetros o valores null accidentales como lo menciona TutorialsPoint.
El Builder moderno resuelve estos problemas permitiendo construcción paso a paso con setters encadenables, asignando solo los campos necesarios del JSON de entrada mientras establece valores por defecto seguros para campos críticos (estado="ACTIVO", fechaDevolucionEsperada=14 días). Los atributos no especificados permanecen null de forma controlada, demostrando la flexibilidad para campos opcionales sin comprometer la integridad del objeto.

PATRÓN PROTOTYPE
El patrón Prototype es un patrón de diseño creacional que permite crear nuevos objetos a partir de la clonación de una instancia existente, en lugar de instanciarlos directamente desde cero. Este enfoque resulta especialmente útil cuando la creación de objetos es costosa o compleja, ya que permite reutilizar configuraciones ya definidas. Los objetos implementan un método de clonación que devuelve una copia de sí mismos, lo que facilita generar múltiples instancias similares sin depender de clases concretas. Según Refactoring.Guru, este patrón permite copiar objetos sin acoplar el código a sus implementaciones específicas, favoreciendo así la flexibilidad y reutilización. Asimismo, GeeksforGeeks señala que es útil cuando la creación directa de objetos implica un alto costo en términos de recursos o complejidad.



•	Código implementado
 
 
 
 

•	Justificación de la implementación
Se implementó el patrón Prototype en las recomendaciones debido a la necesidad de generar múltiples instancias similares a partir de una estructura base común, evitando la reconstrucción repetitiva de objetos complejos. Este enfoque sigue el principio descrito por Refactoring.Guru, donde el patrón Prototype permite copiar objetos existentes en lugar de crearlos desde cero, reduciendo la carga de inicialización y el acoplamiento del sistema.
En este contexto, las recomendaciones comparten una configuración inicial, lo que permitió definir una “recomendación plantilla” preconfigurada que actúa como prototipo. A partir de esta, se generan nuevas instancias mediante clonación, optimizando la eficiencia en la creación de objetos y manteniendo consistencia estructural entre ellas (GeeksforGeeks).
El uso del patrón Prototype también facilita la personalización dinámica, ya que cada clon puede modificarse posteriormente para adaptarse a las características específicas del usuario, agregando información contextual como preferencias, historial o comportamiento. Esto elimina la necesidad de instanciar objetos desde cero o de depender de múltiples constructores complejos, favoreciendo una construcción más flexible, escalable y conforme a las mejores prácticas recomendadas en la literatura sobre patrones de diseño (GeeksforGeeks).

PATRÓN ADAPTER
El patrón Adapter es un patrón de diseño estructural que permite que clases con interfaces incompatibles puedan trabajar juntas, actuando como un intermediario que traduce una interfaz en otra que el cliente espera. Su objetivo principal es reutilizar código existente sin modificarlo, adaptando una clase ya implementada a un nuevo contexto. En este patrón, el adapter envuelve una clase existente (adaptee) y expone una interfaz compatible con el cliente, facilitando la integración entre componentes que originalmente no estaban diseñados para colaborar. Según Refactoring.Guru, este patrón permite que objetos con interfaces incompatibles colaboren sin alterar su código fuente, promoviendo así el desacoplamiento y la flexibilidad en el diseño.

•	Código implementado
 
 
 
 

•	Justificación de la implementación
Se implementó el patrón Adapter en el módulo de suscripciones debido a la necesidad de integrar una clase existente, SubscriptionManager, que ya encapsula la lógica de negocio, sin acoplar directamente al cliente a esta implementación concreta. En lugar de exponer dicha clase, se definió una interfaz de negocio (SubscriptionChecker) que actúa como contrato estable para el cliente (SubscriptionService), garantizando así un diseño desacoplado y alineado con el principio de inversión de dependencias. Este enfoque responde a lo descrito por Refactoring.Guru, donde el patrón Adapter permite que clases con interfaces incompatibles colaboren mediante un intermediario que traduce las llamadas sin modificar el código original.
En este contexto, SubscriptionManagerAdapter cumple el rol de adaptador al recibir las solicitudes a través de la interfaz SubscriptionChecker y delegarlas internamente a SubscriptionManager, específicamente mediante la invocación de su lógica de verificación de suscripciones. Esto permite reutilizar completamente la funcionalidad existente sin necesidad de refactorizar la clase adaptee (SubscriptionManager), preservando su integridad y reduciendo riesgos en el sistema.

PATRÓN BRIDGE
El patrón Bridge es un patrón de diseño estructural que permite separar una abstracción de su implementación, de modo que ambas puedan evolucionar de forma independiente. En lugar de utilizar herencia para cubrir todas las combinaciones posibles (lo que genera muchas subclases), este patrón divide el sistema en dos jerarquías: una para la abstracción (lo que el cliente usa) y otra para la implementación (cómo se realiza). Ambas se conectan mediante composición, lo que permite cambiar o extender cualquiera de las dos sin afectar a la otra. Según Refactoring.Guru, el Bridge evita la proliferación de clases y mejora la flexibilidad del sistema, mientras que SourceMaking señala que este patrón facilita el desacoplamiento entre interfaz e implementación, permitiendo que ambas varíen de manera independiente.

•	Justificación de la implementación
Se implementó el patrón Bridge en el módulo de gestión de libros con el fin de desacoplar la abstracción del libro (LibroBridge) de su implementación de acceso (Acceso), permitiendo que ambas dimensiones varíen de forma independiente. En este caso, la abstracción define operaciones como descarga o tipo de acceso, delegando su comportamiento a implementaciones concretas como AccesoGratis y AccesoPremium, mientras que las especializaciones de libro (LibroPdfBridge, LibroEpubBridge, etc.) representan la variación por formato. Este enfoque, como lo plantean Refactoring.Guru y SourceMaking, evita la proliferación de clases combinatorias y reduce el acoplamiento entre componentes.
En este contexto, LibroBridge actúa como el puente al mantener una referencia a Acceso y delegar en él las operaciones, permitiendo combinar dinámicamente distintos formatos con distintos niveles de acceso (por ejemplo, PDF premium o EPUB gratis) sin crear nuevas clases para cada combinación. De esta manera, el patrón Bridge mejora la flexibilidad, facilita la extensión del sistema y permite incorporar nuevas variantes sin modificar el código existente.

•	Prueba del patrón BRIDGE
Se evidencia el funcionamiento del patrón Bridge en el sistema, donde la abstracción LibroBridge delega la lógica de acceso en la interfaz Acceso, evitando depender directamente de implementaciones concretas. El puente se materializa al combinar dinámicamente el formato del libro (PDF, EPUB, MOBI) con el tipo de acceso (Gratis o Premium), permitiendo que ambas jerarquías evolucionen de manera independiente. La prueba demuestra que, para un mismo formato, el comportamiento cambia según el acceso asignado (por ejemplo, la capacidad de descarga), lo que valida que la responsabilidad está correctamente desacoplada. De esta forma, se confirma que el sistema mantiene consistencia en sus resultados mientras reutiliza la lógica de acceso sin alterar la estructura de los formatos, evidenciando una correcta aplicación del patrón.
 

PATRÓN DECORATOR
El patrón Decorator es un patrón de diseño estructural que permite agregar nuevas funcionalidades a un objeto de forma dinámica sin modificar su clase original. En lugar de utilizar herencia para extender el comportamiento, este patrón utiliza composición, envolviendo el objeto original dentro de uno o varios “decoradores” que añaden responsabilidades adicionales. Cada decorador implementa la misma interfaz que el objeto base, lo que permite encadenarlos y combinar comportamientos de manera flexible. Según Refactoring.Guru, el Decorator permite añadir funcionalidades a objetos de forma transparente para el cliente, mientras que SourceMaking destaca que este patrón es una alternativa más flexible que la herencia para extender comportamientos, evitando la creación de múltiples subclases.

•	Código implementado
 
 
 
 
 
 
 

•	Justificación de la implementación
Se implementó el patrón Decorator en el módulo de notificaciones debido a la necesidad de añadir múltiples canales de envío (como Email, SMS y WhatsApp) de forma flexible y dinámica, sin modificar la clase base NotificadorBasico. Un enfoque tradicional basado en herencia habría generado una proliferación de clases para cada combinación posible de notificaciones, afectando la escalabilidad y mantenibilidad del sistema. En su lugar, se optó por encapsular cada funcionalidad adicional en decoradores independientes, permitiendo extender el comportamiento del objeto en tiempo de ejecución. Este enfoque, según Refactoring.Guru, permite agregar responsabilidades a los objetos de manera transparente, evitando la rigidez de la herencia.
En la implementación, cada decorador (EmailDecorator, SmsDecorator, WhatsappDecorator) actúa como una capa que envuelve al objeto base, interceptando la operación enviar para delegar primero en el componente original y luego añadir su comportamiento específico. Esto permite construir distintas combinaciones de notificación según el contexto (por ejemplo, usuarios gratis vs premium) reutilizando los mismos componentes sin duplicar lógica. Además, el patrón mejora la extensibilidad del sistema, ya que nuevos canales pueden incorporarse sin modificar las clases existentes, cumpliendo con el principio Open/Closed. De esta manera, el patrón Decorator no solo optimiza la composición de funcionalidades, sino que también garantiza un diseño desacoplado, flexible y alineado con buenas prácticas de ingeniería de software.
•	Diagrama UML
 

•	Prueba del patrón DECORATOR
Se evidencia el funcionamiento del patrón Decorator en el sistema, donde el objeto base NotificadorBasico es extendido dinámicamente mediante decoradores como EmailDecorator, SmsDecorator y WhatsappDecorator, sin modificar su implementación original. Cada decorador actúa como una capa adicional que intercepta la llamada y añade un nuevo canal de notificación, delegando primero en el componente base y luego ejecutando su comportamiento específico. La prueba demuestra que, dependiendo del tipo de usuario, se construyen diferentes combinaciones de notificación (por ejemplo, solo email para usuarios gratis y múltiples canales para usuarios premium), validando así la composición dinámica de responsabilidades. De esta manera, se confirma que el sistema mantiene un comportamiento consistente mientras extiende su funcionalidad de forma flexible, cumpliendo con el principio de abierto/cerrado y evidenciando una correcta aplicación del patrón.
